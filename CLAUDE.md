# Monorepo layout (npm workspaces)

- **Three workspaces**: `shared/` (`@metaverse/shared`), `backend/` (`metaverse-backend`), `frontend/` (`frontend`), wired by the root `package.json` `workspaces` field against a **single root `package-lock.json`** (no per-package lockfiles). Install once from the repo root: `npm install`.
- **`@metaverse/shared` is the single source of truth for every wire shape.** All socket event payloads (client→server and server→client), REST request/response shapes, event-name constants, and shared limits (chat length, rate windows, id/coord bounds) are defined **only** in `shared/src` as zod schemas + inferred types. The backend imports the schemas and `safeParse`s them; the frontend imports the inferred **types** (erased at build — zod never enters the bundle) plus the runtime event-name/limit constants. **Never re-declare a payload shape in `backend/` or `frontend/`** — add it to `shared/`. How-to for a new event end-to-end: root `README.md`.
- **Build order**: consumers resolve `@metaverse/shared` from its built `dist/`, so **shared must be built first** (`npm run build:shared`). The root `build`/`typecheck`/`test` scripts and every CI job do this before touching a consumer. `shared/**` and the root `package.json`/`package-lock.json` are in both CI workflows' path filters.
- **Backend Docker image build context is the repo ROOT** (not `backend/`), because the image builds the `shared` workspace from source. `backend/Dockerfile` is built with `-f backend/Dockerfile .`; compose `build` blocks and the deploy workflow set `context: .` / `--file backend/Dockerfile`. The runtime install is scoped (`npm ci --omit=dev -w metaverse-backend -w @metaverse/shared --include-workspace-root`) so no frontend deps leak into the image.

# Production topology

- **Frontend**: hosted on Vercel, auto-deployed to production by the `deploy` job in `.github/workflows/frontend-ci.yml` on every push to `main` (after lint/typecheck/test/build/budget pass). Requires `VERCEL_TOKEN`/`VERCEL_ORG_ID`/`VERCEL_PROJECT_ID` repo secrets. **The frontend imports `@metaverse/shared`, so the Vercel project's Root Directory must be the repository root** — the root `vercel.json` installs the workspace and builds shared before the frontend build.
- **Backend**: a single EC2 box running Docker Compose (`deploy/docker-compose.prod.yml`). `Backend deploy` builds/pushes an immutable ECR image and runs `deploy/deploy-remote.sh` via SSM. The deploy script gates on the migrate+seed `setup` container's exit code before switching the backend image, and alerts + rolls back on health-check failure.
- **Alerting**: a `alerter` container on the box watches the Docker events stream and posts container crashes/restart-loops/unhealthy states to Telegram (`TELEGRAM_BOT_TOKEN`/`TELEGRAM_CHAT_ID` from SSM). See `deploy/README.md`.
- **Version stamps**: the running git SHA is visible in the app's Settings panel (`build <sha>`, injected via `VITE_GIT_SHA`) and in the backend `/health/live` + `/health/ready` responses (`sha`, baked into the image via the `GIT_SHA` build arg).

# Logging conventions

- **Backend**: never use `console.*` in `backend/src` — log through pino (`backend/src/logger.ts`). Create per-module child loggers with `childLogger({ module: "..." })`; inside request handlers use the request-bound logger (`res.locals.log`, or `requestLog(res, fallback)` from `request-logger.ts`) so lines carry the `requestId`; socket code logs through the per-connection child logger (bound with `socketId`, and `playerId`/`spaceId` after join). Pass errors as `{ err }` so pino serializes the stack. `LOG_LEVEL` env controls verbosity (`debug` in dev, `info` in prod).
- **Frontend error beacon**: `frontend/src/errorBeacon.ts` ships uncaught errors/unhandled rejections to backend `POST /client-errors` (installed in `main.tsx`, real-backend mode only). Reports appear in backend logs as `module: "client-error"` with the client's build `sha`. Rate-limited server-side (10/min/IP) and client-side (session cap + dedupe) — telemetry must never break the game.
- **Rotation**: all compose services use the `json-file` driver, `max-size: 10m` × `max-file: 3`, via the shared `x-logging` anchor. Add it to any new compose service.

# Compiler standard (repo-wide, non-negotiable)

- **All three packages compile strict.** Every tsconfig project in the repo — shared (`shared/tsconfig.json`, `shared/tsconfig.eslint.json`), backend (`backend/tsconfig.json`) and all frontend projects (`frontend/tsconfig.app.json`, `tsconfig.node.json`, `tsconfig.e2e.json`, `tsconfig.test.json`) — has `strict`, `noUncheckedIndexedAccess`, and `exactOptionalPropertyTypes` on. Never turn a flag off, and never add a new tsconfig (or override) that compiles code under a weaker standard. `tsc -b` gates the build in CI, so a violation fails the pipeline.
- **Test code is held to the same bar as product code.** Unit tests (`frontend/src/**/*.test.ts(x)`) are strict-typechecked via `frontend/tsconfig.test.json` (wired into `tsc -b`), e2e specs via `tsconfig.e2e.json`. Both packages lint their test files with `@vitest/eslint-plugin` (`no-focused-tests` **errors** — a committed `.only` can never silently shrink CI; `expect-expect`, `no-conditional-expect`, `no-standalone-expect` error; `no-disabled-tests` warns). Write new tests to the same standard: no type escapes, and a `.skip` needs a justification comment (see each README's *.skip-justification convention*). The backend now has ESLint too — `typescript-eslint` type-checked baseline on `src` + `test`, with `no-console` erroring in `src` (logger module exempt), mechanically enforcing the pino convention above.
- **TypeScript stays in lockstep across packages** (same major — currently `~6.0.x` in `shared`, `frontend`, and `backend`). Bump them together and re-verify `typescript-eslint` compatibility; don't let the three diverge.
- **No new `!` (non-null assertion) without a justifying comment.** Prefer a narrowing guard, an early return, or a type derived from the net/contract layer. A genuinely-unavoidable assertion (e.g. a Phaser lifecycle guarantee) must carry a one-line comment saying why it's safe. Frontend **production (non-test) source under `src/`** is currently assertion-free — keep it that way. Test files are strict-typechecked too (unit tests via `tsconfig.test.json`, e2e via `tsconfig.e2e.json`): guaranteed test-harness values like `window.__testHook` use a throwing guard, not `!` — follow that pattern rather than reintroducing bare assertions.

# Backend test conventions

- Unit vs integration split: `npm test` (backend) must stay service-free; `npm run test:integration` requires Postgres + Redis (`docker compose up -d postgres redis`), uses Redis logical DB 1, and runs files sequentially.
- Testability seams: boot the real app via `createApp()`/`createServer()` (src/app.ts) on an ephemeral port — never call route/socket handlers directly; config assertions go through the pure `parseConfig(env)` (src/parse-config.ts); the migration runner takes an overridable migrations dir.
- Isolation: integration tests must never assume clean state — per-run usernames (deleted afterwards), `flushDb` on the dedicated Redis DB, and throwaway Postgres schemas for migration tests.

# Frontend game-logic conventions (scene-as-glue + pure modules)

- **Enforced convention for all new gameplay code (a direction, not a completed state):** game rules go in pure modules under `frontend/src/game/*.ts` (and `media/mediaLogic.ts`) — plain values in, plain values out, **no Phaser / net / DOM imports** — each landing with its vitest file in the same commit. `WorldScene.ts` is the orchestrator: asset/sprite/tilemap setup, Phaser+net event wiring, and per-frame calls into those modules. Putting *new* decision logic in the scene is a review smell — extract it. See `frontend/README.md` → *Scene-as-glue + pure modules* for the module map.
- Extracted and tested today: `movement`, `zones` (containment + room-exit detection), `seatDoor`, `interaction` (interact-key priority), `interpolation`, `throttle`, `proximity` (distance falloff), `audioZones` (room rects → audio zones + zone-aware volume), `interactables`, `maps` (game) and `mediaLogic` (media — room names, track attach/surface/detach routing, zone-aware proximity volumes). Extend these rather than duplicating. **Known logic still in the scene** (extraction candidates — don't grow the list): locked-room position rollback, portal payload validation/teleport, Tiled object parsing, chat-bubble rendering, camera locate, world-info snapshot.
- **Zone-audio convention (no voice through walls).** World voice is proximity audio gated by *audio zone*: `game/audioZones.ts` derives one zone per room interior from the map's `roomBounds` rectangles (zone id = `roomId`) and treats everything else as the single `OUTDOOR_ZONE`; `zoneVolume` returns 0 across zones and the normal `proximity` falloff within a zone (binary cutover at the doorway — no muffling). Zones are computed client-side from already-broadcast positions (each player's `zone` is stamped onto the `positions` payload) — **no wire-format change, no server involvement**. Consequence for map/gameplay work: **any new enclosed area that should be aurally private must be authored as a `roomBounds` rect with a `roomId`** — that (and only that) auto-creates its audio zone; there is no second registry to update. Enforcement is client-side in v1 (documented trust caveat + server-enforced phase-2 path in `frontend/README.md` → *Audio model*).
- The LiveKit split is transport (`livekit.ts`, thin, untested beyond types) vs logic (`mediaLogic.ts`, pure, tested); the transport's track handlers must route through `subscribeAction`/`unsubscribeAction` (`wireTrackRouting`) — no inline attach-vs-surface branches.
- Test style: table-driven / transition-matrix (incl. illegal transitions) for pure modules; React Testing Library + jsdom for React and the `App.tsx` media-transition chain — stub the Phaser canvas + heavy HUD children and assert media-manager calls + rendered HUD state, never Phaser internals or private fields. Everything runs in the single `npm test` (vitest/jsdom) step — no new pipeline stages.
- The `game/eventBus.ts` typed bus is the Phaser↔React contract seam (and the future E2E hook); assert scene→React interactions through it.

# Meeting lifecycle (PRD 10)

- **The meeting-start rules live in exactly one place: the pure trigger state machine `backend/src/meeting.ts`** — every player in the room zone seated ∧ count ≥ 2 arms a cancelable countdown (canceled by a stand or an unseated entry); the countdown elapsing starts the meeting; a latecomer's sit joins in place; a participant's stand (or grace-expired disconnect) leaves alone; the last leaver ends it. Change meeting-start behavior **there**, with its exhaustive transition tests (`backend/test/meeting.test.ts`) — never re-derive the rules in socket handlers or the frontend.
- `backend/src/meeting-manager.ts` is the side-effect shell (countdown timer, per-room serialization, room-scoped broadcasts). The frontend only *reacts*: `frontend/src/game/meetingUi.ts` maps the broadcast events onto client actions (portal-in / portal-out), and `frontend/src/game/portalHandoff.ts` aligns the two-phase portal reveal. Meeting wire payloads live in `@metaverse/shared` like every other event. Client-facing rules + resource policy: `frontend/README.md` → *Meetings*.

# Mini-game architecture (arcade cabinets, PRD 11)

- **Game rules are pure modules under `frontend/src/game/arcade/` — one per game (`snake.ts`, `flappy.ts`, `game2048.ts`) + a shared seeded PRNG (`prng.ts`).** Same enforced convention as all game logic: plain values in/out, **no Phaser/net/DOM imports**, deterministic given a seed + input script (randomness flows through a serializable `rngSeed` in each state, never `Math.random`). Each module lands with its table-driven + determinism vitest in the same commit. **Never put game rules in a React renderer or in `WorldScene`** — the renderer is a thin surface that runs the reducer and draws its output.
- **The overlay + game renderers live in `frontend/src/ui/arcade/` and are lazy-loaded** (`ArcadeOverlay` via `React.lazy`), so the game code is a separate chunk and never enters the entry bundle (bundle-budget gate). Renderers implement `ArcadeGameProps` (`seed`/`paused`/`onScore`/`onGameOver`). The world scene **sleeps** under the overlay (same pattern as meetings) and wakes on `close-arcade`; Escape closes instantly; the overlay pauses on visibility/focus loss and holds keyboard focus robustly (blur stale inputs — the room-key-field focus trap).
- **Games stay audio-agnostic**: they emit domain events on `eventBus` (`arcade-point`/`arcade-over`/`arcade-flap`; `open-arcade` opens the overlay), and the sound mixer's event→clip table decides the blip — never call audio from game logic.
- **Cabinets are placed via the campus generator** (`scripts/gen_campus.py`: a solid `furn(...)` sprite + an `interactType="arcade"` interactable carrying a `game` payload), never by hand-editing `campus.json`; sprites come from `scripts/gen_arcade_sprites.py` (+ BootScene key + ATTRIBUTIONS row). **High scores are one REST resource** with shapes in `@metaverse/shared` (`arcade*Schema`), best-per-user in Postgres, rate-limited — **client-reported and trusted at this level** (documented caveat; no server-side play validation in Phase 1). The canonical game-id list is `ARCADE_GAMES` in shared. How-to (module contract): `frontend/README.md` → *Arcade cabinets*.

# Board-game tables (two-player, server-authoritative, PRD 11 phase 2)

- **Board tables are a distinct subsystem from arcade cabinets: two-player, server-authoritative, and NOT arcade/REST.** Arcade is single-player + client-trusted REST scores; board tables (tic-tac-toe, Connect-4) are turn-based matches the **server owns end-to-end** — never trust the client's board. Don't route board logic through the arcade overlay, the `interactables` layer, or the private-room seat system.
- **The game rules are ONE pure implementation in `shared/src/games/` (`board.ts`, `ticTacToe.ts`, `connect4.ts`, `rules.ts`) — imported by BOTH the backend (authoritative validation) and the frontend (rendering + click→move mapping).** Plain values in/out, deterministic, zero deps; `applyMove(state, player, index)` returns the next state or a typed `BoardMoveError`. Exhaustive win/draw/illegal-move matrices land in `shared/src/games/*.test.ts`. **Never re-derive board rules** in backend or frontend — extend the shared module. `rulesFor(game)` resolves a game id to its `create`/`applyMove` functions; the `BOARD_TABLES` registry in `shared/src/constants.ts` is the single source of truth for which tables exist and which game each runs.
- **The match lifecycle lives in exactly one place: the pure machine `backend/src/boardMatch.ts`** (both seats ⇒ offer; both accept ⇒ start; stand/leave ⇒ forfeit, offer ⇒ cancel; win/draw ⇒ over; both empty ⇒ waiting), modeled on `meeting.ts` with exhaustive transition tests (incl. illegal transitions) in `backend/test/boardMatch.test.ts`. `backend/src/board-manager.ts` is the side-effect shell (per-table serialized dispatch, disconnect-grace forfeit timer, Redis snapshot with TTL, space-scoped `board-update` broadcasts + per-client `board-error`), modeled on `meeting-manager.ts`. **Change match behavior there, not in socket handlers or the frontend.** Live match state (board, turn, participants, game) is mirrored to Redis at `board:<tableId>` with a TTL (add prefixes to `resetEphemeralGameState`).
- **Turns are server-validated socket events** (`board-sit`/`board-stand`/`board-accept`/`board-move`), safeParsed against strict zod schemas in `shared/src/socket.ts`; seat membership, turn order and move legality are checked against the shared rules and illegal/out-of-turn moves get a typed `board-error` (the client is never trusted). Every payload is defined in `@metaverse/shared` like every other wire shape.
- **Frontend: board tables reuse the seat *mechanics* (map-authored seats, sit visuals, `seatDoor`) but a SEPARATE `board_seats` map layer + `WorldScene.boardSeats` array** — they are public plaza seats, ungated by room entry, so they never trip the meeting trigger or the minimap room list. The world does **not** sleep — players stay seated in-world while a lazy-loaded HUD panel (`ui/BoardTablePanel.tsx`) floats over the board; spectators walking up to a live table see the same panel read-only. Client-side decisions (view model, whose turn, offer prompt, spectator display, click→move) live in the pure `game/boardTable.ts` (+ vitest). Sounds go through the `soundMixer` event→clip table (`board-sat`/`board-move`/`board-win`, reusing existing clips) — game logic stays audio-agnostic. How-to (adding a table game): `frontend/README.md` → *Board-game tables*.

# Graphics + sound direction (PRD 12)

- **Art & audio direction is locked — keep style cohesion.** Top-down pixel art in the current **Pipoya-compatible** family: no resolution/HD-2D re-theme, no avatar redraws, no music composition. Furniture is a cohesive cool office palette. Tilesets are **re-skinned/enriched, never re-laid-out** — map coordinates, collision rects, `roomBounds` and seats are preserved, and `frontend/src/game/maps.test.ts` cross-checks every tileset against its on-disk PNG (dimensions, columns, tilecount) plus GID validity. **Reject a style-mismatched asset rather than blending it.** Audio bar: fewer, better, normalized clips — no filler.
- **Sound decisions live in exactly one pure module: `frontend/src/media/soundMixer.ts`** (channels master→music/sfx/ambient, master mute, ambient duck near voice, the event→sound table, footstep cadence), tested with playback mocked. `media/sfx.ts` is thin HTMLAudio glue (gain application, music/ambient loops, autoplay-unlock); `ui/SfxBridge.tsx` is headless event→clip wiring. **Game logic stays audio-agnostic** — it emits domain events on `eventBus`, the bridge decides the sound. Volumes persist via `ui/settings.ts`. Engine-side ambience (day/night tint, particles, sway) is **code, not assets** (`game/dayNight.ts` + WorldScene).
- **Every shipped asset must have a row in `frontend/ATTRIBUTIONS.md`** (source, author, license) — unattributed assets must not ship. Curate (don't bulk-import) from the packs in the untracked repo-root `Assets/`; NEVER commit raw zips/packs. Audio regenerates via `frontend/scripts/gen_audio.py` (Ogg Vorbis, normalized). Asset-pipeline how-to: `frontend/README.md` → *Sound + polish pipeline*. The bundle-budget CI gate measures only the gzipped entry JS chunk, not `public/` assets; raise it consciously in `scripts/bundle-budget.mjs` (with a comment) if code polish ever demands it, never silently.

# E2E conventions (Playwright)

- The E2E suite lives in `frontend/e2e/` and runs chromium-only against the BUILT frontend (`vite preview`, port 4173) + the docker-composed backend stack. It is PR-blocking (`e2e` job in `frontend-ci.yml`). Full local-run recipe: `frontend/README.md` → *E2E tests (Playwright)*.
- **Assert through the bus hook + DOM HUD only.** `window.__testHook` (`frontend/src/e2e/testHook.ts`) exposes the event bus and minimal game state; it exists only in builds made with `VITE_E2E_HOOK=1` and must stay tree-shaken out of production bundles (CI greps the prod dist for `__testHook`). Never read pixels off the Phaser canvas, and never assert network internals already covered by the backend/smoke suites.
- **No arbitrary sleeps.** Every wait is a bus event or DOM condition (`waitForFunction` on hook state, `expect(locator)`). Movement waits ride the `positions` tick. If a scenario flakes, fix the wait condition — do not add sleeps or bump retries (CI retries stay at 1, local at 0).
- Scenarios run serially (`workers: 1`): all tests share live space "1", so parallel workers physically interfere. Each test signs up a fresh user (the room-key rate limit is per player+room, keeping tests isolated; the auth limiter is per-IP — restart the backend container if local iterating hits it).
- LiveKit assertions stop at "token fetched, connection attempted" with fake media devices (chromium flags in `playwright.config.ts`); media quality is out of scope.
- Map waypoints in `frontend/e2e/helpers.ts` are verified straight-line segments against the walls layer + solid furniture; re-verify against the map JSON when the maps change.
