1. The Experience

  • Open space — players move on a tilemap; proximity audio turns on as avatars get near (audio only out here, no video). Gather-style spatial audio.
  • Private rooms — key-gated. Walk to a room's door zone → prompted for a key → correct key lets you in.
  • Inside a room — 4–5 seats. Sit down → full video + audio call with others seated.
  • The twist — the 2D environment stays visible during the call. In-room webcams render as bubbles floating above each seated avatar, not a Meet-style panel.

Two media contexts (LiveKit)

  • World room — one LiveKit room everyone joins, audio tracks only. The frontend scales each remote participant's volume by 2D distance (from Phaser positions). Beyond a cutoff radius → volume 0. This is the proximity audio.
  • Private rooms — each private room is its own LiveKit room with audio + video, joined only after a correct key + sitting in a seat; left on standing up.

4. The Contract SHARED — this is what Codex builds to

4a. REST (HTTP API)

Method & PathBody / AuthReturns
POST /api/v1/signup{username, password}200 ok / 400 if taken
POST /api/v1/signin{username, password}{ token }
GET /api/v1/space/:idBearer token{ mapJsonUrl, rooms:[{id,name,doorZone,seats:[{id,x,y,facing}]}] }
POST /api/v1/livekit/tokenBearer token, {roomName}{ livekitToken, url }

Notes: token authenticates both REST and the Socket.IO handshake. Room keys are validated server-side only — never trust the client. The LiveKit token must be scoped to the requested roomName and the caller's identity.

4b. Socket.IO events

// client → server
join         { token, spaceId }
move         { x, y, dir }                 // dir: 'down'|'left'|'right'|'up'
chat         { text }
room-enter   { roomId, key }               // server validates key
seat-sit     { roomId, seatId }
seat-stand   { }

// server → client
init          { selfId, players:[{id,name,x,y,dir}] }      // sent once after join
player-joined { id, name, x, y, dir }
player-moved  { id, x, y, dir }                            // broadcast to space
player-left   { id }
chat          { id, name, text, scope }                    // scope:'world'|roomId
room-enter-result { ok, roomId, reason? }                  // reason: 'bad-key'|'full'
seat-update   { roomId, seatId, playerId|null }            // occupied / freed

Movement is broadcast per space; for scale use Socket.IO rooms (one per space, and optionally one per private room for chat scoping). Throttle inbound move server-side if needed; the client already throttles to ~10–15/s and interpolates remotes.

4c. Shared types — contract.ts

The frontend keeps the canonical event/payload types in frontend/src/contract.ts. Codex should mirror these exactly (copy the file or generate matching types). Any change is a joint change.

5. Backend Responsibilities Codex

Services

  • HTTP API — auth (token issue/verify), space fetch, LiveKit token minting.
  • Socket.IO server — movement broadcast, presence, chat, key validation, seat state.
  • LiveKit SFU — self-hosted; the "Meet architecture" for in-room A/V and world audio. Backend mints scoped access tokens; LiveKit handles the media.

Data model (Postgres)

  • users(id, username, password_hash)
  • spaces(id, name, map_json_url)
  • rooms(id, space_id, name, key_hash, door_zone, capacity)
  • seats(id, room_id, x, y, facing)

Redis: presence + (later) multi-WS-server fan-out so people in one space share a sticky server. Single server is fine for the first ~10k users / handful of rooms.

6. Asset Pipeline Frontend

  • Packs unzipped into frontend/public/assets/ (tilesets, characters, furniture, doors, audio).
  • Primary tileset: TopDownHouse (Retro Interior), 16×16 tiles — clean, cozy, Zep-like. Characters: Pipoya 32×32 (sheets are 96×128 = 3 cols × 4 rows → down/left/right/up walk).
  • Map authored in Tiled → exported to public/assets/maps/space.json; layers: ground, walls (collision), furniture; object layers doorZones (prop roomId) and seats (props roomId, seatId, facing).

8. Deployment (single t3.medium)

  • Nginx reverse proxy + Let's Encrypt TLS (WSS required for Socket.IO + WebRTC in prod).
  • Node HTTP + Socket.IO under PM2. Postgres + Redis local or managed.
  • LiveKit self-hosted on the box (CPU-only is fine for a few small video rooms). Frontend only needs the token endpoint + ws URL. Add a TURN/STUN config for NAT traversal.
  • 2 GB swap recommended (Node + WS + LiveKit can spike memory on t3.medium).
